# SaaS-Builder — Autonomous Product & Marketing Strategist

A multi-agent orchestration system that turns a raw SaaS idea into a complete, internally-consistent launch bundle: a technical architecture, a 12-week go-to-market roadmap, and an HTML dashboard — generated by a pipeline of cooperating LLM agents.

Built with **Python**, **OpenAI GPT-4o-mini**, and **Streamlit**.

![Agent pipeline architecture](p116.svg)

## Why

Early-stage SaaS ideas usually fail for the same reason: developers start writing code before they've nailed down who the product is for, how it makes money, or whether the technical plan can actually support the marketing promises. This project automates that missing strategy layer by simulating a small "virtual team" — a product manager, an architect, a marketer, and a critic — that interrogates the idea, plans it, builds both sides of it in parallel, and checks that the two sides agree with each other before calling it done.

## How it works

The system is not a single prompt — it's a pipeline of specialized agents (`agent.py`), coordinated by an orchestrator (`orchestrator.py`), each responsible for one part of the process:

1. **Inverse Prompting Agent** — instead of guessing what the user means, it asks clarifying questions to pin down the SaaS niche, target users, and monetization model before any planning starts.
2. **Planning Agent** — turns the clarified idea into a structured, step-by-step build plan.
3. **Parallel execution** — the plan is fanned out to two agents that run **concurrently** using Python's `threading`:
   - **Technical Architecture Agent** → produces a JSON spec (tech stack + database schema)
   - **Marketing Strategy Agent** → produces a Markdown 12-week go-to-market roadmap
4. **Consistency Agent** — reads both outputs together and flags/fixes conflicts, so the marketing plan never promises a feature the technical design doesn't support.
5. **Critic / Reflection Agent** — reviews the aligned bundle and produces feedback for "industrial readiness"; this loop runs for a configurable number of iterations (`REFLECTION_ITERATIONS`, default 2).
6. **Dashboard Renderer** (`dashboard_renderer.py`) — assembles the final JSON architecture and Markdown roadmap into a single, styled HTML dashboard (`output/saas_bundle_dashboard.html`).

Along the way, `metrics.py` tracks LLM call count, total tokens, and latency for every agent call, so the computational cost of a full run is visible rather than hidden.

## Two ways to run it

- **`streamlit_app.py`** — the interactive version. Enter an idea → the app generates clarifying questions → you answer them → it runs the full pipeline (`run_pipeline`) and renders the plan, architecture, roadmap, reflection rounds, live metrics, and the embedded HTML dashboard directly in the browser.
- **`run.py`** — a CLI version. Enter an idea at the prompt and it runs the non-interactive pipeline (`run_system`), printing the architecture, roadmap, and critique straight to the terminal.

## Tech stack

| Layer | Tool |
|---|---|
| LLM | OpenAI GPT-4o-mini (`llm.py`) |
| Orchestration | Python `threading` for parallel agent execution |
| Web UI | Streamlit |
| Dashboard rendering | Custom HTML/CSS + `markdown` library |
| Metrics | Custom lightweight tracker (`metrics.py`) — LLM calls, tokens, latency |

## Project structure

```
saas-builder-multi-agent/
├── agent.py                # All individual agent functions (inverse prompting, planning, technical, marketing, consistency, critic)
├── orchestrator.py          # Wires the agents into the two pipelines (run_system, run_pipeline)
├── llm.py                   # Thin wrapper around the OpenAI Chat Completions API
├── metrics.py                # Tracks LLM calls / tokens / latency per run
├── logger.py                 # Lightweight run logger; records agent calls and renders a flow-diagram SVG

├── dashboard_renderer.py     # Renders the final architecture + roadmap into an HTML dashboard

├── run.py                    # CLI entry point
├── main.py                   # Project metadata/definition
├── p116.d2 / p116.svg         # Architecture diagram (source + rendered)
├── output/                   # Generated dashboards and run graphs land here
└── requirements.txt
```

## Getting started

### Prerequisites
- Python 3.11+
- An OpenAI API key with billing enabled — each full pipeline run makes ~6+ LLM calls (gpt-4o-mini) and will incur usage cost on your own key.

### Installation

```bash
git clone https://github.com/itu-itis22-karai22/saas-builder-multi-agent.git
cd saas-builder-multi-agent
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### Set your API key

`llm.py` reads the key from the environment, so set it before running:

```bash
# macOS / Linux
export OPENAI_API_KEY="your-key-here"

# Windows (PowerShell)
$env:OPENAI_API_KEY="your-key-here"
```

### Run the interactive app

```bash
streamlit run streamlit_app.py
```

### Or run it from the terminal

```bash
python run.py
```

## Sample output

A full run produces:
- A **JSON technical architecture** (tech stack + database schema)
- A **Markdown 12-week go-to-market roadmap**
- One or more **critic/reflection passes** refining the bundle
- An **HTML dashboard** (`output/saas_bundle_dashboard.html`) combining the architecture and roadmap into a single visual view
- **Run metrics**: total LLM calls, total tokens used, and end-to-end latency

## Known limitations

- **LLM-dependent quality**: since every agent's output comes from GPT-4o-mini, the depth and accuracy of the strategy is bounded by the underlying model.
- **Sequential foundation**: agents run in parallel *after* planning, but the Planning Agent itself runs once and sequentially — if it misses a key requirement, all downstream agents inherit that gap.
- **Token cost**: the reflection loop and dual parallel agents mean a single "SaaS bundle" costs noticeably more tokens than a single-shot prompt would.
